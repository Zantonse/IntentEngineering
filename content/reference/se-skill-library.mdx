## Design Patterns Quick Reference

### Pattern 1: Static vs. Dynamic Context

```yaml
# Static: product knowledge in references/
See [references/product-capabilities.md]

# Dynamic: account data via injection
Account context: !`python scripts/crm_fetch.py $ARGUMENTS`
```

Use static references for stable institutional knowledge (capabilities, certifications, pricing). Use dynamic injection for data that changes per account вЂ” CRM records, org charts, deal history.

---

### Pattern 2: Stakeholder-Aware Output

```
For CTOs: Lead with business outcomes and ROI
For engineers: Include API examples and data models
For procurement: Emphasize TCO, contract flexibility
```

Apply in: `technical-proposal`, `rfp-analyzer`, `demo-builder`, `solution-architect`.

---

### Pattern 3: Calibrated Confidence

| Label | Meaning |
|---|---|
| **Full** | Meets requirement without configuration |
| **Partial** | Met with configuration or third-party integration |
| **Planned** | On confirmed roadmap with delivery date |
| **Gap** | Not supported, no current plan |

Apply in: `rfp-analyzer`, `security-questionnaire`, `technical-proposal`.

---

### Pattern 4: Auto-Invocation Control

| Setting | When to Use |
|---|---|
| `disable-model-invocation: true` | Customer-facing skills: RFP, proposal, email вЂ” require human review before output is sent |
| Omit or `false` | Background knowledge skills: `objection-handler`, product-knowledge вЂ” safe to invoke automatically |

---

### Pattern 5: Context Fork for Research

```yaml
context: fork
agent: Explore
```

Use for deep research skills (`account-brief`, `win-loss-analysis`, `battlecard-builder`). Keeps verbose exploration out of the main context window. Results are summarized and returned to the parent context on completion.

## Naming Conventions

| Convention | Pattern | Examples |
|---|---|---|
| Action skills | Verb-noun, lowercase with hyphens | `rfp-analyzer`, `demo-builder`, `poc-manager` |
| Knowledge skills | Noun, lowercase with hyphens | `product-knowledge`, `competitive-intel` |
| Length | 2вЂ“3 words max | Avoid `technical-discovery-preparation-tool` |
| Character limit | 64 characters max on the `name` field | Enforced by Claude Code skill loader |

## Priority Order for Building

| Priority | Skill | Rationale |
|---|---|---|
| 1 | `discovery-prep` | Highest frequency вЂ” runs before every new opportunity |
| 2 | `rfp-analyzer` | Highest time savings вЂ” RFP response is the most time-intensive recurring task |
| 3 | `battlecard-builder` | Highest competitive impact вЂ” win rate effect is immediate and measurable |
| 4 | `security-questionnaire` | Most repetitive вЂ” same questions answered repeatedly with identical approved language |
| 5 | `demo-builder` | Most customization value вЂ” generic demos lose deals that tailored demos win |
